------------------------------------------------------------------------------
YE PROJECT HAI KYA BHAI?
------------------------------------------------------------------------------

Toh bhai, ye ek "Caution Money Refund System" ka backend hai — basically ek
server jो college ke students ko unka caution deposit wapas dilaata hai.

Jab tum college join karte ho, college kuch paise "security deposit" ke naam
pe leta hai — jise caution money bolte hain. Aur jab tum college chhod do
(padhke ya bhaagke, dono case mein), toh wo paise wapas milne chahiye.

Problem? College ka system itna slow hai ki paise lene ke liye bhi
5 alag-alag departments se "haan bhai theek hai" karwana padta hai.
Isliye ye system banaya — taaki sab digitally ho aur koi excuse na bache.

Ye backend ye sab handle karta hai:
  - Student ka registration aur login
  - Bank details deke refund application submit karna
  - Document upload step — student 3 images upload karta hai: TC/PG admission
    slip, bank passbook front page, aur fees slip. Teeno ek hi request mein,
    ek saath save hote hain. Evidence collect karo, paisa baad mein.
  - 5 departments se clearance lena (library, sports, hostel, dept, accounts)
  - Har department ka apna admin login
  - Admins ke liye dashboard — kaun sa application kitne mein hai
  - Offline no-dues mode — student physically form sign karwake photo upload
    kar sakta hai, admin image dekh ke approve karta hai. Ek form, ek image,
    sab departments ka ek saath kaam. Digital aur physical dono — jugaad nahi,
    feature hai.


------------------------------------------------------------------------------
KAM KAISE KARTA HAI? (Bade picture mein)
------------------------------------------------------------------------------

Socho backend ek "middleman dost" hai jo frontend (website) aur database ke
beech mein baitha hai. Frontend bolti hai "ye karo" — backend check karta hai
"sahi hai?" — database se data laata hai — aur response bhejta hai.

Simple flow:

  1. Frontend request bhejti hai  →  "Student login karna chahta hai"
  2. Backend check karta hai      →  "Password sahi hai? Fake toh nahi?"
  3. Database se baat karta hai   →  Student ka data fetch/save karo
  4. Response bhejta hai          →  "Le bhai tera token" ya "galat password"

Student ka poora safar:
  Register → Login → Bank Details Daal → Application Submit →
  Documents Upload karo (TC/Admission Slip + Passbook + Fees Slip) →
  Clearances ka Wait karo (aur karo... aur karo...) → Paise wapas!

Admin ka kaam:
  Login → Applications dekho → Apne department ki clearance approve/reject karo

------------------------------------------------------------------------------
EK REQUEST KA SAFAR (Step-by-step — application submit karne ka example)
------------------------------------------------------------------------------

Student ne "Submit Application" button dabaaya. Ab kya hoga?

  1. Frontend POST request bhejti hai → /api/application/submit

  2. Rate Limiter check karta hai → "Bhai 15 minute mein 100 se zyada
     requests mat maar, spam mat kar"

  3. Auth Middleware → "Tera JWT token hai? Valid hai? Expired toh nahi?
     Haan sab theek? Chalo aage."

  4. Validator → "Bank account number diya? IFSC sahi format mein hai?
     Declaration checkbox tick kiya? Sab sahi? Achha."

  5. Controller → "Service bhai, handle karo isko"

  6. Service → "Dekho — kya ye student pehle se apply kar chuka hai?
     Nahi? Theek hai, application banao."

  7. Repository → "MongoDB, ye nayi application save karo please"

  8. Response → "Application submitted successfully! ID: IEHE-XYZ123"

  Student: "Finally!"  Backend: "Mera kaam ho gaya, ab clearances ka wait karo."

USERS KAUN HAIN? (Roles)
------------------------------------------------------------------------------

Do type ke log hain is system mein:

  STUDENTS
    - Apne enrollment number se register karte hain
    - Login karke refund application submit karte hain
    - Apna clearance status track karte hain
    - Basically wait karte hain aur F5 dabaate rehte hain

  ADMINS (6 types — har ek ka apna raaj)
    - superadmin    → Sabka baap. Full access. Jo chahe kar sakta hai.
    - library       → "Kitaabein wapas di? Haan? Cleared."
    - sports        → "Sports kit return ki? Ground ka kiraya baaki nahi?
                       OK cleared."
    - hostel        → "Hostel room saaf chhoda? Mattress nahi churaaya?
                       Cleared."
    - department    → Apne department ke students ki clearance
    - accounts      → Sabka favourite — ye wala refund actually process karta hai
                       (toh inhe khush rakhna)


------------------------------------------------------------------------------
APPLICATION KI LIFE (Ek application ka poora safar)
------------------------------------------------------------------------------

  submitted          → Student ne form bhara. Ab wait.
       ↓
  under_review       → Admins dekh rahe hain. Ab aur wait.
                       (Is stage pe student online ya offline choose kar sakta hai)
                         ONLINE  → Admin seedha portal se clearance deta hai
                         OFFLINE → Student physically no-dues form sign karwata hai,
                                   ek photo upload karta hai, admin image dekh ke
                                   portal se approve karta hai. Ek form, sab ke
                                   signs — library, sports, hostel, dept — ek
                                   jagah. One image to rule them all.
       ↓
  partially_cleared  → Kuch departments ne haan bola, baaki abhi soch rahe hain
       ↓
  on_hold            → Kisi ne "hold" maar diya — kuch issue hai
                       (library ki book missing? Sports ka racket nahi diya?)
       ↓
  fully_cleared      → Sabne approve kar diya! Accounts ko forward!
       ↓
  refund_processed   → Paisa aaya bhai! Transaction ID bhi mili!

  (rejected)         → Kisi bhi stage pe reject ho sakta hai. Life hai.

Kaun si clearances chahiye:
  - Library    → Hamesha chahiye (books return karo pehle)
  - Sports     → Hamesha chahiye
  - Department → Hamesha chahiye
  - Hostel     → Sirf hostellers ke liye (day scholars — free pass!)
  - Accounts   → Last step — ye process karenge actual refund

------------------------------------------------------------------------------
TOOLS AUR TECHNOLOGIES (Aur kyun use kiye)
------------------------------------------------------------------------------

SERVER CHALANE KE LIYE
  Node.js
    JavaScript sirf browser mein chalta tha pehle. Node.js ne bola "bhai
    server pe bhi chalao" — aur duniya badal gayi. Yahan isi pe server hai.

  Express.js
    Node.js ke upar ek framework. Bina iske har route ke liye khud se
    HTTP parsing likhni padti. Express ne easy kar diya — "bas router
    banao aur chal pado."

DATABASE
  MongoDB
    Traditional databases mein data fixed tables mein rehta hai (Excel
    jaisa). MongoDB mein flexible documents hain — nested clearances,
    timelines, sab ek hi document mein fit ho jaata hai. Hamare use case
    ke liye perfect.

  MongoDB Atlas
    MongoDB ka cloud wala version. Database kisi aur ki machine pe hai
    (Amazon AWS ke servers pe), hamari local machine pe nahi. Free tier
    mein kaam chala liya — broke student energy.

  Mongoose
    MongoDB se directly baat karna rough hota hai. Mongoose ne ORM
    banaya — schema define karo, validations lagao, aur .save() likho
    bas. Kaam ho gaya.

LOGIN / SECURITY KE LIYE
  JWT (jsonwebtoken)
    Login hone pe server ek "token" deta hai — ek signed digital ID card
    jaisi cheez. Har request pe ye token bhejo, server samajh jaata hai
    "haan ye banda legitimate hai." Server ko kuch yaad nahi rakhna — token
    mein sab info hoti hai. Clean aur scalable.

  bcryptjs
    Passwords kabhi seedha database mein mat save karo — ye basics hai.
    bcrypt password ko hash (scramble) kar deta hai. Agar kal database
    bhi hack ho jaaye, attacker ke haath sirf "xK$2b$12$..." milega —
    actual password nahi. Salt rounds 12 matlab extra secure.

  helmet
    Ek line ka kaam — app.use(helmet()) — aur 10+ security headers
    automatically lag jaate hain. Content-Security-Policy, X-Frame-Options,
    sab. Basically ek security wala bodyguard free mein.

  cors
    "Kaun kaun baat kar sakta hai hamare server se?" — ye cors decide
    karta hai. Sirf hamare frontend URL ko allow kiya hai. Random
    websites directly API call nahi maar sakti.

  express-rate-limit
    "Bhai 15 minute mein 100 se zyada request mat maar." Spam aur
    brute-force attacks se bachne ke liye. Auth routes pe aur bhi
    strict — 10 attempts hi milte hain.

FILE UPLOAD KE LIYE
  multer
    Students image upload karte hain offline no-dues ke liye. Multer ye sab
    handle karta hai — incoming multipart/form-data request parse karo, file
    ko memory mein rakho (disk pe nahi — Vercel pe disk read-only hai bhai),
    size aur type check karo. memoryStorage use hota hai taaki Vercel jaisi
    serverless environments pe bhi kaam kare. req.file.buffer mein milta hai
    file — seedha MongoDB tak bhejo bina disk chhue.

  sharp
    Image compression library. Upload hoti image ko resize aur compress karta
    hai (max 1600x1200, quality 75%) — taaki database mein 5MB raw image
    save karne ki zaroorat na ho. Agar compression fail ho toh original buffer
    seedha save ho jaata hai — graceful fallback. sharp native binaries use
    karta hai toh fast bhi hai.

INPUT CHECK KARNE KE LIYE
  express-validator
    User ne jo bheja wo sahi format mein hai ya nahi — ye check karta
    hai. IFSC code galat? Mobile number 11 digit ka? Enrollment number
    mein special characters? Sab pakad leta hai pehle hi.

  dotenv
    .env file se secret values load karta hai code mein. Database
    password seedha code mein likhoge toh GitHub pe push hoga aur
    duniya padh legi. dotenv ne ye problem solve ki.

DEVELOPMENT KE LIYE
  morgan
    Har incoming request terminal mein print karta hai —
    "GET /api/health 200 3ms" — debugging mein bahut kaam aata hai.
    "Aayi request? Gayi? Kitne time mein?" — sab dikh jaata hai.

  nodemon
    Code save kiya, server automatically restart. Bina iske har change
    ke baad Ctrl+C aur npm run dev — too much pain. nodemon developer
    ki jaan hai.

  prettier
    Code format karta hai automatically. Tab ya spaces ki ladai khatam.
    Semicolons ka scene khatam. Sab consistent dikhta hai — team mein
    koi bhi padhega toh readable lagega.


------------------------------------------------------------------------------
DESIGN DECISIONS (Ye kyun banaya aisa — justify karo bhai)
------------------------------------------------------------------------------

CONTROLLERS / SERVICES / REPOSITORIES ALAG KYUN?
  Socho agar sab ek hi file mein hota — database query, business logic,
  HTTP response — toh ek bug dhundna matlab poori file padhna. Alag karne
  se agar bug hai toh pata hai kahan dhundna hai. Controller mein? Request
  issue. Service mein? Logic issue. Repository mein? DB issue. Simple.

JWT KYUN, SESSIONS KYUN NAHI?
  Sessions mein server ko yaad rakhna padta hai "ye user logged in hai."
  Multiple servers ho toh problem. JWT mein token khud hi sara data
  carry karta hai — server stateless rehta hai. Frontend-backend alag
  hain hamare — JWT fit baithta hai perfectly.

MONGODB KYUN, SQL KYUN NAHI?
  Application document mein clearances nested hain, timeline array hai,
  bank details embedded hain. SQL table mein ye sab joins se laana padta
  — complex aur slow. MongoDB mein ek document mein sab kuch — ek query
  mein poori application. Our data is document-shaped, toh document DB
  chose kiya.

ROLE-BASED ACCESS KYUN?
  Library admin ko refund process karne ka kya kaam? Accounts admin ko
  library clearance approve karne ka kya kaam? Roles ne clear boundaries
  bana di hain. Galti se bhi koi kuch aur nahi kar sakta.

STATUS AUTOMATIC COMPUTE KYUN?
  Manually overallStatus update karna matlab ek aur jagah bug ho sakti
  hai. Pre-save hook automatically calculate karta hai — clearances update
  karo, status khud update ho jaayega. DRY principle — Don't Repeat Yourself.
